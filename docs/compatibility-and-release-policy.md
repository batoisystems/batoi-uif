# Compatibility and Release Policy

## Browser Baseline

Batoi UIF targets the current and previous two stable major releases of Chrome, Edge, Firefox, and Safari, plus iOS Safari 16.4 and later. The distributed JavaScript target is ES2022. Progressive capabilities such as service workers, push, install prompts, WebSocket, Server-Sent Events, observers, and Trusted Types remain feature-detected; their absence must not break the server-rendered baseline.

Accessibility behavior is tested in the DOM unit suite and must be verified with keyboard and assistive technology during release browser QA. Content editing, selection, clipboard, composition, layout, and mobile interactions require real-browser verification because jsdom is not an engine-conformance substitute.

## Public API Stability

Public package exports, documented `data-uif-*` attributes, lifecycle events, and generated browser globals are the compatibility surface for v2.

- Additive APIs and attributes may ship in minor releases.
- Behavior fixes that preserve documented intent may ship in patch releases.
- A deprecated API remains available for at least one minor release and is documented before removal.
- Removal or incompatible signature, event, attribute, or markup-contract changes require a major release.
- Internal source modules, generated chunk names, editor-private source markers, and undocumented CSS internals are not stable APIs.

## Lifecycle Contract

Stateful initializers own their listeners, timers, observers, subscriptions, and requests. They return either a controller with `refresh()` and `destroy()` or an explicit disposer. Repeated initialization returns or replaces existing ownership without stacking behavior. `destroy()` is idempotent and cancels pending work.

Stateless decorators may return `void` when repeated application only sets the same classes or ARIA attributes and creates no retained listener, observer, timer, subscription, or request.

## Release Budgets

`release-budgets.json` is enforced by `npm run verify:release`.

- Root ESM JavaScript and CSS artifacts must remain below 120,000 gzip bytes. The global IIFE artifact has a format-specific 120,100-byte ceiling to account for its required wrapper; this does not raise the modular runtime budget.
- Package JavaScript budgets include every emitted `.js` chunk in that package.
- The jsdom component initialization smoke path must complete within 50 ms on at least one of three runs.
- Framework packages must have no third-party runtime dependencies.

Budget increases require an explicit review of product value, runtime cost, and zero-dependency alternatives. Generated output must be rebuilt and compared before release.

The IIFE allowance was introduced after router/RAD security hardening. Implementation deduplication reduced the measured IIFE from 120,389 to 120,055 gzip bytes before the 100-byte format allowance was accepted; no package budget was raised.

`dist/integrity.json` is regenerated by `npm run build:dist` and records SHA-256, SHA-384 SRI, byte, and gzip values for every root distribution artifact. Release verification rejects stale, missing, extra, or modified root artifacts. Consumers serving fixed-version browser files may use the recorded `sri` value in an `integrity` attribute together with the deployment's appropriate `crossorigin` policy.

`release-api.json` schema v2 records public export names and deterministic SHA-256 hashes of normalized exported declarations. Release verification rejects missing declarations, removed exports, and changed function/interface/type/variable signatures. Additive exports remain allowed. Intentional signature changes or removals require the documented major-version process. Regenerate the baseline with `npm run release:api-baseline` only after reviewing public API changes.

`npm run verify:packages` runs `npm pack --dry-run --json` for every framework package. It verifies that declared exports, main/module entries, and declarations exist in the actual publish set; wildcard exports resolve; and source, tests, specs, and source maps are not shipped. `npm run verify:release` includes this check.

`npm run verify:packed-install` creates tarballs for every framework package, installs the complete package graph into a fresh temporary project in offline and ignore-scripts mode, and imports every JavaScript package through its public package entry point. Temporary artifacts are removed after success or failure. This catches missing internal dependencies, broken export maps, and package-only resolution failures that workspace tests cannot expose.

## Release Gate

Required automated checks are lint, unit tests, full build, release verification, browser tests, and `git diff --check`:

```bash
npm run lint
npm test
npm run build
npm run verify:release
npm run test:browser
npm run audit:security
git diff --check
```

The Playwright matrix covers Chromium, Firefox, WebKit, and a representative narrow Chromium viewport. A public release additionally requires manual assistive-technology, CSP/Trusted Types, and deployed-artifact checks.

Development requires Node.js 20.19 or later in the Node 20 line, or Node.js 22.12 or later. The patched Vite 8, Vitest 4, jsdom 29, and Playwright toolchain is development-only and does not alter framework runtime dependencies. npm currently reports one low-severity Windows development-server advisory through the esbuild range shared by Vite and tsup; do not expose local development servers to untrusted networks, and remove this note when those tools accept a patched esbuild release.
